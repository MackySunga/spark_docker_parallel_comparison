Traditional Parallel Processing vs Apache Spark in Docker

This simulation compares a regular parallel processing model with fixed worker threads against an Apache Spark style model running inside Docker worker containers. The goal is to show why Spark becomes powerful on large or skewed workloads: it partitions data, dynamically schedules work, tolerates stragglers better, and coordinates distributed execution across executors.

Interpretation Guide

Use this as a teaching simulation, not as a hardware benchmark. It illustrates the logic of parallel scheduling and distributed execution, especially how Spark partitions work and how Docker packages the cluster environment.

  • Traditional Parallel is usually good when the job is small, the machine is local, and the chunks are balanced.
  • Spark in Docker shines when workloads are bigger, more uneven, or need scalable orchestration across several worker containers.
  • Docker adds deployment isolation and portability, but it does not create extra compute power on its own. Containers still share the host resources unless distributed across multiple machines.
  • More Spark partitions are not always better. Too many partitions increase scheduling and shuffle overhead.
  • High work skew is where dynamic scheduling matters the most because it reduces idle waiting caused by one overloaded worker.
Teaching tip: try a low dataset size with low skew first, then increase both. Students can observe how startup overhead may make Spark look slower for tiny jobs, while larger and more imbalanced jobs often favor the Spark model.

Compact Equations Used in the Simulation

These simplified equations are educational approximations designed to visualize behavior.

Traditional Parallel: T_trad ≈ T_start + max(thread_load_i / thread_rate) + T_merge Imbalance_trad ≈ stdev(thread_loads) / mean(thread_loads) Utilization_trad ≈ busy_thread_time / (threads × elapsed_time) Apache Spark in Docker: T_spark ≈ T_container_start + T_driver_sched + max(executor_load_j / exec_rate) + T_shuffle_reduce Imbalance_spark ≈ stdev(executor_loads) / mean(executor_loads) Speedup (Spark over Traditional) ≈ T_trad / T_spark
  • T_start is the ordinary program startup overhead.
  • T_container_start models Docker worker startup and Spark initialization.
  • T_driver_sched represents the driver and master scheduling cost.
  • T_shuffle_reduce is the final combine stage across partitions.
  • exec_rate and thread_rate are the effective work rates of a processing slot.
